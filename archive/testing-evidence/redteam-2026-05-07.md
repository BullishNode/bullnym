# Red-team run against bullpay.ca — 2026-05-07

Historical test report. Some implementation names and rollout references may
no longer match current server code.

Aggressive end-to-end test pass against the live pay-service from a disposable LunaNode VM in the Toronto region (single IP `170.75.171.81`). Goal: probe every documented defense at its boundary; try to break the server. User directive was "go all-out, don't hold back."

Logs: `pay-simulator/test-runs/redteam-20260506/`. Server-side journalctl tail: `redteam-20260506/server-tail.log` (76,295 lines of `event="rate_limited"` warnings captured during the run).

## Headline findings

**Server stayed up under everything we threw at it.** `/health` returned `ok` at every checkpoint — before, during, and after each scenario, including after a single-IP **108,729-request 60-second burst (1,811 rps sustained)**. Not a single 5xx observed end-to-end.

The deployed rate-limit v2 stack is doing its job. Most simulator "FAIL"s are the cap firing during fixture setup — that *is* the test passing.

## Production hardening findings (`sec-headers` tier)

These are **real, fixable defects** in nginx/axum response headers — minor severity, but worth filing:

| ID | Finding | Fix |
|---|---|---|
| **SEC-01** | `Server: nginx/1.24.0 (Ubuntu)` leaks exact version | Set `server_tokens off;` in nginx config |
| **SEC-03** | No `Strict-Transport-Security` header on HTTPS responses | Add `add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;` to nginx |
| **SEC-09** | `OPTIONS` returns `Access-Control-Allow-Headers: *` — echoes any attacker-requested header | Pin to a literal whitelist |
| **SEC-12** | No `Permissions-Policy` header | Add `add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;` |

(SEC-06, SEC-13 also failed but only because the per-IP register cap blocked their fixture setup; they'd likely PASS with rotation.)

## Single-IP attack ceiling

After this run we have a numbered ceiling for single-IP attackers:

- **At most 3 fresh nyms can be registered per IP per hour** (`register_distinct_npubs_per_ip_limit=3`). Sustained nym-creation requires real IP rotation.
- **At most 5 register requests per IP per minute** (`register_rate_limit=5`).
- **At most 60 callback requests per IP per minute** (`per_ip_limit=60`). Sustained 1,811 rps from one IP yielded ~61 successful through-puts in 60s = exactly the rate-limit budget.
- **At most 30 metadata probes per IP per minute** (`metadata_rate_limit=30`).
- **At most 10 webhook posts per IP per minute** (`webhook_rate_limit=10`).
- **At most 30 Lightning callbacks per source per hour** (`lightning_per_source_limit=30`).

A pure single-IP attacker is throttled to a 60-rps ceiling on the callback path and cannot do bulk fresh-nym creation at all.
